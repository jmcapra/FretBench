## 1. Overview

FretBench is a benchmark for evaluating LLM ability to reason about guitar fretboard note names from ASCII tablature. It consists of:

1. **A curated test-case dataset** (private, versioned separately)
2. **A local CLI** for orchestrating evaluation runs against LLMs via OpenRouter
3. **An SQLite results database** for tracking all runs
4. **A static website** (Astro on Cloudflare) displaying leaderboard and detailed results

The website is 100% static — no server-side logic, no database queries at runtime. All dynamic data is baked into the site at build time from the SQLite database.

## 3. Test Case Dataset

### 3.1 Repository Structure

- **Public repo:** `jmcapra/FretBench` — all code, website, CLI
- **Private repo:** `jmcapra/fretbench-test-cases` — test case data only
- Tracked as a **git submodule** at `./test-cases/`
- The file `test-cases/test-cases.json` is the canonical dataset path
- The root-level `test-cases.json` is legacy and must be removed / gitignored

### 3.2 Dataset Format

```json
[
  {
    "id": "FB_001",
    "tuning": "Standard",
    "tab": "e|---0---|\nB|---1---|\nG|-------|\nD|-------|\nA|-------|\nE|-------|",
    "question": "What is the note being played on the B string?",
    "answers": ["C"],
    "strict_spelling": false
  }
]
```

- **100 test cases** across 4 tunings: Standard (33), Half-Step Down (34), Drop D (16), Drop Db (17)
- `answers`: array of acceptable responses (handles enharmonic equivalents like `["C#", "Db"]`)
- `strict_spelling`: when `true`, only exact matches from `answers` are accepted

### 3.3 Versioning

- Dataset is versioned via git tags in the private repo (e.g., `v1.0.0`, `v1.1.0`)
- Every benchmark run records the dataset version (git commit SHA + tag if present)
- The submodule pin in the main repo tracks which dataset version is active

### 5.3 Prompt Templates

#### System Prompt (eval version 1.0.0)

```
You are a guitar theory expert. You will be shown a snippet of ASCII guitar tablature and asked to identify a specific note.

## How to read guitar tablature

- Each line represents a guitar string, labeled on the left (e.g., `e`, `B`, `G`, `D`, `A`, `E`).
- Numbers on a string indicate the fret to be played. `0` means the open string.
- The note sounded is determined by the string's tuning pitch plus the fret number (each fret = one semitone up).
- Notes are read left to right in chronological order.
- Dashes (`-`) are spacing and produce no sound. Pipes (`|`) are bar lines.
- Letters between fret numbers (e.g., `h`, `p`, `/`, `\`) are articulation marks (hammer-on, pull-off, slide) — they do not change which notes are played.

## Tuning reference

The tab's string labels tell you the tuning. Common tunings and their open-string pitches (thinnest → thickest):

- **Standard:** e B G D A E → E4 B3 G3 D3 A2 E2
- **Half-Step Down:** eb Bb Gb Db Ab Eb → Eb4 Bb3 Gb3 Db3 Ab2 Eb2
- **Drop D:** e B G D A D → E4 B3 G3 D3 A2 D2
- **Drop Db:** eb Bb Gb Db Ab Db → Eb4 Bb3 Gb3 Db3 Ab2 Db2

## The chromatic scale (ascending from C)

C → C#/Db → D → D#/Eb → E → F → F#/Gb → G → G#/Ab → A → A#/Bb → B → C

## Instructions

1. Identify the correct string from the question.
2. Find the fret number(s) played on that string in the tab.
3. Starting from the string's open pitch, count up by the fret number in semitones using the chromatic scale.
4. Respond with ONLY the note name (e.g., `F#`). No explanation, no other text.
```

#### User Prompt Template

```
{{tab}}

{{question}}
```

Where `{{tab}}` is the raw ASCII tab string (newlines preserved) and `{{question}}` is the question text verbatim from the test case. No tuning name is provided — the model must infer it from the string labels in the tab.

#### Example: Assembled User Prompt

```
e|-------------------|
B|-------------------|
G|-------------------|
D|-0-h-3---5---0-----|
A|-0-h-3---5---0-----|
D|-0-h-3---5---0-----|

What is the second note played on the D (6th) string?
```

#### Design Rationale

- **No tuning name in the user prompt.** The string labels (e.g., `eb Bb Gb Db Ab Eb` vs `e B G D A E`) already encode the tuning. Recognizing the tuning from string labels is part of what the benchmark tests — stating it explicitly would bypass that reasoning step.
- **System prompt includes tuning reference and chromatic scale.** The benchmark tests *reasoning* ability, not memorization of tuning definitions. Providing the reference pitches and chromatic scale is the equivalent of giving a human test-taker a reference sheet — the hard part is still reading the tab, recognizing the tuning from string labels, counting semitones, and mapping to the right note.
- **"ONLY the note name"** — forces concise output, reduces extraction ambiguity, keeps token cost minimal.
- **Articulation marks explicitly called out** — prevents models from being confused by `h`, `p`, `/` characters between fret numbers.
- **No few-shot examples** — avoids leaking patterns that could inflate scores. The system prompt teaches the *skill*, not specific answers.

#### Versioning

Any change to the system prompt, user prompt template, temperature, or max_tokens constitutes a new eval version. The full system prompt text is stored in the `eval_versions` table (see §6.2), and its SHA-256 hash enables quick drift detection.

### 5.4 Run Orchestration

1. **Load dataset** from `test-cases/test-cases.json`
2. **Record dataset version** (git SHA of the submodule HEAD, plus tag if tagged)
3. **Resolve eval version** — hash the current system prompt and compare against the latest `eval_versions` row. If it differs, the CLI prompts to create a new eval version before proceeding.
4. **For each test case**, assemble the user prompt from the template and send to the model via OpenRouter:
   - System prompt: as defined above
   - User prompt: rendered template
   - Temperature: `0` (deterministic)
   - Max tokens: `64` (answers are single notes)
5. **Grade the response** using the grader (see §5.6)
6. **Record results** to SQLite (see §6)
7. **Track token usage and cost** from API response headers/body

### 5.5 Parallel Execution

- When running `--all` or `--tier`, models are evaluated in parallel
- Concurrency is capped (default: 3 models simultaneously) to respect OpenRouter rate limits
- Configurable via `--concurrency N` flag
- Within a single model run, requests are sequential (one test case at a time) to avoid per-model rate limits
- Rate limit errors (429) trigger exponential backoff with jitter

### 5.6 Grading Logic

```
function grade(response: string, testCase: TestCase): boolean
  1. Normalize: trim whitespace, strip surrounding quotes/periods
  2. Extract note name: find the first token matching /^[A-Ga-g][#b]?$/
  3. If strict_spelling is true:
       return answers.includes(extracted) (case-insensitive)
  4. If strict_spelling is false or absent:
       return answers.includes(extracted) (case-insensitive)
       OR enharmonicEquivalent(extracted) is in answers
```

Enharmonic map: `C#↔Db, D#↔Eb, F#↔Gb, G#↔Ab, A#↔Bb`

### 5.7 Cost Tracking

For every API call, record from the OpenRouter response:
- `prompt_tokens` (input token count)
- `completion_tokens` (output token count)
- `total_cost` (as reported by OpenRouter in the `usage` object or `x-openrouter-cost` header)

**Pre-run cost estimation (`--dry-run`):**
- Estimate input tokens by tokenizing the system prompt + each test case prompt (use a rough `chars / 4` heuristic or `tiktoken` if available)
- Estimate output tokens as ~10 per response (single note name)
- Look up per-model pricing from the model registry or OpenRouter API (`GET /api/v1/models`)
- Display: estimated total cost, cost per question, input vs output breakdown

## 6. Database Schema (SQLite)

### 6.1 File Location

`data/fretbench.db` — gitignored, lives locally only.

### 6.2 Tables

```sql
-- Tracks eval methodology versions: system prompt, grading logic, parameters.
-- Bumped whenever anything changes that could affect comparability of results.
CREATE TABLE eval_versions (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  version         TEXT NOT NULL UNIQUE,    -- semver, e.g. "1.0.0", "1.1.0"
  system_prompt   TEXT NOT NULL,           -- full system prompt text
  system_prompt_hash TEXT NOT NULL,        -- SHA-256 of system prompt for quick comparison
  grading_logic   TEXT NOT NULL,           -- description or version of grading algorithm
  temperature     REAL NOT NULL,           -- e.g. 0.0
  max_tokens      INTEGER NOT NULL,        -- e.g. 64
  notes           TEXT,                    -- free-text changelog ("tightened extraction regex")
  created_at      TEXT NOT NULL            -- ISO 8601
);

-- One row per benchmark run (one model, one pass through the full dataset).
CREATE TABLE runs (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  model_id        TEXT NOT NULL,           -- OpenRouter model ID
  eval_version_id INTEGER NOT NULL REFERENCES eval_versions(id),
  dataset_version TEXT NOT NULL,           -- git SHA of test-cases submodule
  dataset_tag     TEXT,                    -- git tag if present (e.g. "v1.0.0")
  started_at      TEXT NOT NULL,           -- ISO 8601
  completed_at    TEXT,                    -- ISO 8601
  status          TEXT NOT NULL DEFAULT 'running' -- running | completed | failed
);

-- One row per test case per run. This is the core data table.
-- All aggregates (score, cost totals, tuning breakdowns) are derived from here.
CREATE TABLE run_results (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  run_id          INTEGER NOT NULL REFERENCES runs(id),
  test_case_id    TEXT NOT NULL,           -- e.g. "FB_001"
  tuning          TEXT NOT NULL,           -- e.g. "Standard", denormalized for query convenience
  question        TEXT NOT NULL,           -- the question asked
  expected        TEXT NOT NULL,           -- JSON array of acceptable answers
  strict_spelling BOOLEAN NOT NULL DEFAULT 0,
  raw_response    TEXT,                    -- full model response text
  extracted       TEXT,                    -- note name extracted by grader
  correct         BOOLEAN NOT NULL,        -- did extracted match expected?
  prompt_tokens   INTEGER,                -- input tokens for this call
  completion_tokens INTEGER,              -- output tokens for this call
  cost            REAL,                    -- USD for this single API call (from OpenRouter)
  latency_ms      INTEGER,                -- wall-clock response time
  error           TEXT                     -- error message if this call failed
);

CREATE INDEX idx_run_results_run_id ON run_results(run_id);
CREATE INDEX idx_run_results_test_case ON run_results(test_case_id);
CREATE INDEX idx_run_results_tuning ON run_results(tuning);
CREATE INDEX idx_runs_model ON runs(model_id);
CREATE INDEX idx_runs_eval_version ON runs(eval_version_id);
```

### 6.3 Derived Aggregates (Views)

All summary statistics are computed from `run_results` — never stored as denormalized columns on `runs`.

```sql
-- Per-run summary (score, cost, tokens)
CREATE VIEW run_summary AS
SELECT
  r.id AS run_id,
  r.model_id,
  r.eval_version_id,
  r.dataset_version,
  r.started_at,
  r.completed_at,
  COUNT(rr.id) AS total_cases,
  SUM(rr.correct) AS correct,
  ROUND(100.0 * SUM(rr.correct) / COUNT(rr.id), 2) AS score_pct,
  SUM(rr.prompt_tokens) AS total_prompt_tokens,
  SUM(rr.completion_tokens) AS total_completion_tokens,
  SUM(rr.cost) AS total_cost,
  ROUND(AVG(rr.latency_ms)) AS avg_latency_ms
FROM runs r
JOIN run_results rr ON rr.run_id = r.id
WHERE r.status = 'completed'
GROUP BY r.id;

-- Per-run, per-tuning breakdown
CREATE VIEW run_tuning_breakdown AS
SELECT
  run_id,
  tuning,
  COUNT(*) AS total,
  SUM(correct) AS correct,
  ROUND(100.0 * SUM(correct) / COUNT(*), 2) AS score_pct
FROM run_results
GROUP BY run_id, tuning;
```

## 7. Summary Statistics

Computed from `run_results` via SQL views/queries. These are **not** additional API calls — they are pure SQL/JS aggregations over existing data.

### 7.1 Per-Model Statistics

| Metric | Description |
|--------|-------------|
| **Overall Score** | % correct across all test cases (latest run) |
| **Mean Score** | Average score across all runs of this model |
| **Std Deviation** | Score variance across runs |
| **Best / Worst Run** | Highest and lowest scores |
| **By Tuning** | Score breakdown: Standard, Drop D, Half-Step Down, Drop Db |
| **By Answer Type** | Score for natural notes vs accidentals (sharp/flat) |
| **Total Cost** | Cumulative spend across all runs |
| **Avg Latency** | Mean response time per question |
| **Token Efficiency** | Avg completion tokens per response |

### 7.2 Cross-Model Statistics

| Metric | Description |
|--------|-------------|
| **Leaderboard Ranking** | Models sorted by latest run score |
| **Hardest Questions** | Test cases with lowest success rate across all models |
| **Easiest Questions** | Test cases with highest success rate across all models |
| **Tuning Difficulty** | Average score per tuning across all models |
| **Provider Comparison** | Average score grouped by provider |
| **Cost vs Performance** | Score plotted against total run cost |

## 11. Non-Goals (Explicit Exclusions)

- No server-side rendering or API routes on the website

- No user accounts, authentication, or public submission of runs
- No hosting of the test dataset in the public repo
- No automatic model discovery — models are manually curated in the registry
